# Default Browser Switcher 要件ドラフト

## 目的

BYOD の MacBook Pro で、仕事時間帯は Microsoft Edge、プライベート時間帯は Safari など、利用シーンに応じて macOS のデフォルトブラウザをすばやく切り替える。

## 想定ユーザー

- 平日日勤帯は仕事用ブラウザを使う。
- 私用時間帯はプライベート用ブラウザを使う。
- URL を別ブラウザへコピーして開く手間を減らしたい。
- BYOD 端末のため、会社管理ポリシーや私用環境への影響を最小化したい。

## 対象環境

- macOS 向けデスクトップアプリ。
- 初期実装の主対象は、現在利用中の MacBook Pro。
- 配布形態は当面ローカル利用を想定する。
- 将来的に署名・公証・自動アップデート対応を検討できる設計にする。

## コア機能

### ブラウザ切り替え

- macOS の HTTP / HTTPS URL のデフォルトハンドラを切り替える。
- 初期プリセットとして Safari と Microsoft Edge を扱う。
- Chrome、Firefox、その他将来登場するブラウザも追加できるよう、ブラウザは固定 enum ではなく bundle identifier ベースで管理する。
- 現在のデフォルトブラウザを表示する。
- インストール済みで HTTP / HTTPS を扱えるアプリを検出して候補として表示する。
- 切り替え実行後、HTTP と HTTPS の両方が同じブラウザになっていることを確認する。

### 手動切り替え

- メニューバー常駐アプリからブラウザを選んで切り替えられる。
- グローバルショートカットキーで、任意のブラウザへ切り替えられる。
- 最低限、以下の操作を用意する。
  - 仕事用ブラウザに切り替え
  - プライベート用ブラウザに切り替え
- ショートカットキーは設定画面から変更できる。
- 他アプリや macOS 標準ショートカットと衝突した場合は設定時に検知または警告する。
- Caps Lock をショートカットに使えるかを優先検討する。
- Caps Lock 単独押下は macOS 標準のホットキー API では扱いにくいため、MVP では Control + Option + E、Control + Option + S を初期ショートカットにする。
- Caps Lock 単独または Caps Lock 併用ショートカットは、アクセシビリティ権限や低レベルキーボードイベント監視が必要になる可能性があるため、技術検証後に対応可否を確定する。

### タイマー切り替え

- 任意の時刻に仕事用ブラウザ / プライベート用ブラウザへ切り替える。
- 「月曜 8:00、火曜 8:00...」のような繰り返し登録ではなく、「稼働日の 8:00」「稼働日の 18:00」のように設定できる。
- 初期定義では稼働日を月曜から金曜とする。
- 日本の祝日は稼働日から除外する。
- 将来的に会社休日、有給、日本以外の国の祝日などの例外日を扱える余地を残す。
- Mac がスリープ中・電源オフ中に予定時刻を過ぎた場合、次回起動・復帰時に現在時刻に合うブラウザへ補正する。
- タイマー機能はオン / オフできる。
- 手動切り替え後も、次回タイマー時刻で自動的に既定ルールへ戻す。

## 設定

- ブラウザプロファイル
  - 表示名
  - bundle identifier
  - 用途ラベル: 仕事用、プライベート用、その他
  - 切り替え単位はブラウザアプリ単位とし、ブラウザ内プロファイル単位の切り替えは扱わない。
- 稼働日
  - 初期値: 月曜から金曜
  - 日本の祝日を除外する。
  - 将来候補: 日本以外の祝日、カレンダー連携、手動例外日
- 稼働時間
  - 初期値候補: 8:00 から 18:00
  - タイムゾーンは macOS の現在設定に従う。
- 起動
  - ログイン時に自動起動する。MVP 必須とする。
  - メニューバーに常駐する。
- 通知
  - 切り替え成功時の通知はオン / オフ可能。
  - 失敗時は通知またはメニューバー表示で知らせる。

## ショートカット初期値

- 仕事用ブラウザへ切り替え: Control + Option + E
- プライベート用ブラウザへ切り替え: Control + Option + S
- トグル切り替えは MVP から除外する。
- Caps Lock 単独押下
  - 使いやすいが、macOS 標準のグローバルショートカットとしては扱いにくい。
  - MVP では初期値にせず、設定画面で利用可能性を検証したうえで将来対応とする。

## 配布・署名方針

- 開発中はローカルビルドで検証する。
- 実利用用ビルドは Developer ID Application 証明書で署名し、Apple の notarization を通す。
- 署名・公証済み `.app` または `.dmg` として配布する。
- Apple Developer Program アカウントがない場合、以下のどちらかを選ぶ。
  - 個人の Apple Developer Program に登録して Developer ID 署名する。
  - 会社の IT 部門に社内配布用の署名・承認・MDM 配布を依頼する。
- 会社が個人署名アプリを許可しない場合は、社内 IT 承認済みアプリとして配布する必要がある。
- MDM でデフォルトブラウザ変更自体が制限されている場合、このアプリ側では回避せず、ポリシー確認を前提にする。

## 非機能要件

- 会社環境でも使いやすいよう、常駐負荷を小さくする。
- ブラウザ切り替え以外の URL 内容、閲覧履歴、Cookie、プロファイル情報は収集しない。
- 設定はローカルに保存する。
- 初期実装ではクラウド同期を行わない。
- macOS の将来バージョンで Launch Services の挙動が変わる可能性を前提に、切り替え処理を小さなモジュールへ隔離する。
- 会社のセキュリティポリシーで未署名アプリが許可されない可能性があるため、配布用ビルドは Developer ID 署名と notarization を前提にする。
- MDM や社内ポリシーで未承認アプリの実行、ログイン項目登録、デフォルトブラウザ変更が制限される場合は、社内 IT への確認または例外申請が必要になる。

## 技術方針案

- ネイティブ macOS アプリとして Swift / SwiftUI で作る。
- メニューバー常駐には `MenuBarExtra` または AppKit の status item を使う。
- 設定ウィンドウを MVP に含める。
- デフォルトブラウザの取得・設定は Launch Services を利用する。
- ログイン時起動は macOS 13 以降では `SMAppService` を利用する。
- スケジュールは常駐アプリ内のタイマーと、起動・復帰時の状態補正で実装する。
- 将来的に確実なバックグラウンド実行が必要になった場合は LaunchAgent 化を検討する。

## MVP

1. インストール済みブラウザ候補の検出。
2. Safari / Microsoft Edge の手動切り替え。
3. メニューバー常駐 UI。
4. グローバルショートカットで仕事用 / プライベート用へ切り替え。
5. ログイン時自動起動。
6. 設定ウィンドウ。
7. ショートカットキー変更。
8. 切り替え失敗時の明確な表示。
9. Developer ID 署名と notarization に対応した配布手順。

## 次バージョン

- 稼働日・開始時刻・終了時刻による自動切り替え。
- 日本の祝日除外。
- 手動切り替え後、次回タイマー時刻で既定ルールへ自動復帰。

## 後回し候補

- 日本以外の祝日カレンダー連携。
- Google Calendar / Outlook Calendar などの業務カレンダー連携。
- ブラウザ別プロファイル指定。
- 一時的な「今日だけ仕事用を延長」設定。
- URL パターンに応じたブラウザ振り分け。
- チーム配布用の署名・公証・MDM 対応。
- Caps Lock 単独押下による切り替え。

## 確認したい論点

1. Apple Developer Program の個人または法人アカウントで Developer ID 署名できるか。
2. 会社の BYOD / MDM ポリシーで、個人 Developer ID 署名アプリの実行が許可されるか。
3. Caps Lock 単独押下を将来対応として扱ってよいか。
